# Changelog

All notable changes to SOURSOP are documented in this file.

## 2.0.3 (July 2026)

A performance and correctness release: multi-model PDB files load roughly twice as fast, and a package-wide bug review fixed a number of correctness defects (several of which silently produced wrong numbers). A few fixes intentionally change numerical output — see **Bug fixes** below.

### Bug fixes
* `ssprotein.get_local_to_global_correlation`: **Fixed an off-by-one that excluded the C-terminal residue from every distance pair** (`range(start, end - seq_sep)` → `range(start, end - seq_sep + 1)`). The terminal residue never entered the local-vs-global correlation and the largest sequence separation contributed *zero* pairs, biasing every reported `mean_corr`/`std_corr`. **This changes the numerical output** of this function; the committed reference values were rebuilt.
* `ssprotein.calculate_all_CA_distances` (`mode='COM'`): **Fixed `stride` being silently ignored** — the COM branch called `get_inter_residue_COM_distance()` without forwarding `stride`, so `get_distance_map(mode='COM', stride=...)` (and `get_polymer_scaled_distance_map`) returned the unstrided result, and combining `mode='COM'` with `stride` *and* `weights` raised a `ValueError` (weights strided to a shorter length than the unstrided data). The `mode='CA'` path was always correct and is unchanged.
* `ssprotein.get_clusters`: **Fixed misuse of `scipy.cluster.hierarchy.ward`** — the full square RMSD matrix was passed directly, which scipy interprets as *observation vectors* (clustering on the Euclidean distance between RMSD-profile rows) rather than as a precomputed distance matrix. The matrix is now symmetrised and condensed with `scipy.spatial.distance.squareform` before clustering, so clustering is performed on the RMSD values as intended. **This changes cluster assignments.** A clear `SSException` is now raised when a stride leaves fewer than two frames to cluster (previously a cryptic scipy error).
* `ssprotein.get_secondary_structure_DSSP`: **Fixed per-residue H/E/C fractions not summing to 1** at residues DSSP cannot classify (mdtraj's `'NA'` code, emitted at chain termini / caps). `'NA'` now folds into coil (coil = neither helix nor strand), so the fractions form a complete partition summing to 1, as documented. **This changes the coil fraction at uncapped termini**; the committed reference values were rebuilt.
* `ssprotein.get_distance_map` (weighted): now returns `None` for the standard-deviation map when `weights` are supplied, as documented, instead of a NaN-filled array (assigning `None` into the float array silently produced NaNs).
* `ssprotein.get_angle_decay`: **Fixed a crash on single-frame trajectories** — a bare `np.squeeze` collapsed the frame axis when `n_frames == 1`, raising `AxisError`. Now squeezes only the length-1 atom axis.
* `ssprotein.get_asphericity`: added the divide-by-zero guard its siblings (`get_acylindricity`, `get_prolateness`) already have, so a degenerate (zero-trace) gyration tensor returns a finite value rather than `NaN` + a `RuntimeWarning`.
* `ssprotein.get_RMSD`: now accepts NumPy integer frame indices (e.g. from `np.arange`/`np.where`); `isinstance(frame2, int)` alone silently rejected `np.int64` and fell through to the compare-vs-all-frames branch.
* `ssprotein.get_Q` (`protein_average=False`): the returned residue-key list is now actually sorted (`np.sort(list)` returned a discarded array; replaced with `sorted(...)`).
* `ssnmr.compute_random_coil_chemical_shifts`: **Fixed a crash on phospho-Ser/phospho-Thr with `use_perdeuteration=True`** — the guard checked residue codes 22/25/28 but the phospho codes are `SEP=20, TPO=21, PTR=22`, so pSer/pThr slipped past and raised `IndexError` on the 20-element deuteration tables instead of the documented `SSException`.
* `ssnmr` (sequence parser): **Fixed an `IndexError` on out-of-range characters** — the bound `code > 35` was tested against a 26-element table, so characters mapping to codes 26–35 (e.g. lowercase `a`–`d`, `[ \ ] ^ _ \``) indexed out of range. Also fixed the amino-acid label list desyncing from the parsed sequence when a character is skipped, which mislabelled and dropped downstream residues.
* `ssdata`: **Fixed a merged residue name** — `"KM2KM3"` (a missing comma) is now `"KM2"` and `"KM3"`, so mono/di-methyl-lysine residues are recognised as valid protein residues rather than silently dropped during chain detection.
* `sstools.find_trajectory_files`: **Fixed replicate-directory matching** — `str.endswith(tuple_of_digit_strings)` matched trailing *characters* rather than the replicate *number*, so e.g. `rep15` was wrongly included for `num_replicates=5` (ends in `'5'`) while `rep7` was dropped. The trailing integer is now parsed and range-checked, so the correct set of replicates is gathered.
* `sspre.SSPRE`: clarified that `W_H` is the **angular** proton Larmor frequency (`omega_H = 2*pi*nu_H`, rad/s), which is what the Solomon–Bloembergen spectral-density term the code evaluates requires and what SOURSOP's shipped label-cloud calibration used. The docstring, example, and sanity-check range previously described `W_H` as the *linear* frequency (e.g. `600000000` for a 600 MHz magnet); following that guidance made the dispersive relaxation term a factor of `(2*pi)^2` too small (overestimating gamma by ~7% at `tau_c = 5` ns). **No numerical behaviour changed** — only the documentation and the validation-warning range/message.
* `sssampling.SamplingQuality`: **Fixed the default excluded-volume reference path crashing on uncapped chains** — the EV reference was gathered once per residue, but `get_angles('phi'/'psi')` returns one fewer angle than there are residues for an uncapped chain (phi undefined at the N-terminus, psi at the C-terminus), so the arrays failed to broadcast. The reference is now trimmed from the appropriate terminus to match.
* `sssampling.SamplingQuality`: **Fixed `truncate=True` crashing for `proteinID > 0`** — truncation rebuilds each trajectory from the single selected protein (now at index 0), but downstream code still indexed with the original `proteinID`, raising `IndexError`. `proteinID` is reset to 0 after truncation.
* `ssbme.BMECustom`: the default chi-squared path now uses its analytic gradient (`jac=True`) instead of a finite-difference gradient. The analytic `_default_cost_grad` was previously dead code and the docstring's claim of an analytic gradient was false; the finite-difference path cost O(N²·m) per optimizer step on the large ensembles the class targets. Custom cost functions still use finite differences. The optimum is unchanged (analytic gradient verified against finite differences to ~2e-9).
* `ssbme.BME`: dual Lagrange-multipliers are now initialised from a seeded local RNG rather than the global `np.random.normal`, so fits are reproducible run-to-run (the convex dual converges to the same optimum regardless, so results are otherwise unchanged).

### Performance
* `sstrajectory` (`__readTrajectory`): Multi-model PDB files — the standard output of idpconformergenerator, PED downloads, and NMR-style entries — now load **~2x faster** by parsing the file exactly once. When the trajectory file *is* the topology file and both are a PDB (i.e. `SSTrajectory(x.pdb, x.pdb)`), SOURSOP now calls `md.load_pdb()` directly rather than `md.load(..., top=...)`. The latter pre-parses the topology via mdtraj's `_parse_topology()` → `load_frame(top, 0)`, but mdtraj's PDB reader has no random access: it reads *every* model and then discards all but the first, so the whole file is parsed twice (once for the topology, once for the coordinates). On a 520 MB, 1000-model, 6.4M-atom tau ensemble this took loading from **58.9 s to 31.8 s**; the profile confirms the redundant pass (12,856,000 = exactly 2 × 6,428,000 `Atom.__init__` calls before the fix). The `pdblead=True` path no longer re-reads the same file a third time either. The returned trajectory is byte-for-byte identical (coordinates, topology, bonds, unit cell), and every other combination of inputs — including the common `xtc` + `pdb` case — takes the original code path unchanged. Note this removes the redundant pass only; the remaining cost is mdtraj's pure-Python parse of every `ATOM` record, so converting a large multi-model PDB once to a binary format (e.g. `.xtc`) remains far faster for repeated analysis.

### Testing
* Added `tests/test_bugfixes_2_0_3.py`, a dedicated regression suite pinning each of the correctness fixes above (l2g terminal-residue inclusion, COM-mode stride, `get_clusters` degenerate-frame guard, single-frame `get_angle_decay`, DSSP fractions summing to 1, weighted std-map `None`, NumPy-int `get_RMSD`, `KM2`/`KM3` validity, `find_trajectory_files` replicate ranges, phospho/out-of-range `ssnmr` handling, uncapped EV-reference sampling, and the `BMECustom` default fit).
* Rebuilt the committed reference-observable pickles (`soursop/data/test_data/*_reference.pkl`) to capture the corrected `get_local_to_global_correlation` and `get_secondary_structure_DSSP` (coil) values. A key-by-key diff confirmed these were the **only** observables that changed.
* Updated `tests/test_ssproteins.py::test_get_distance_map_weights` to assert the corrected weighted std-map contract (`None` rather than a NaN array).
* Added tests in `tests/test_sstrajectory.py` for the multi-model PDB fast path: the predicate that selects it (fires only for the same on-disk `.pdb`; not for `xtc`+`pdb`, differing PDBs, non-PDB files, or `None`), numerical equivalence of the fast path against the old `md.load(pdb, top=pdb)` path (exact coordinates, topology and radius of gyration), and that `pdblead=True` still prepends the PDB frames.

## 2.0.2 (July 2026)

A feature release adding a coarse-grained spin-label model to the PRE calculation, and making it the default.

### Breaking changes
* **`sspre.SSPRE.generate_PRE_profile` now defaults to `use_label=True`.** By default the paramagnetic centre is now modelled as the coarse-grained spin-label cloud described below (calibrated against DEER-PREdict) rather than sitting directly on the `spin_label_atom` (the classic point-at-CB model). This produces different — and, against DEER-PREdict's explicit MTSL rotamer treatment, more accurate — I_para/I_dia and Γ₂ profiles than SOURSOP ≤ 2.0.1. **To reproduce results from earlier versions exactly, pass `use_label=False`.**

### New features and improvements
* `sspre.SSPRE.generate_PRE_profile`: Added a coarse-grained spin-label cloud model, controlled with `use_label` (now `True` by default; see above). Instead of placing the paramagnetic centre directly on the anchor atom, the unpaired electron is modelled as a deterministic cloud of `n_label_conformers` beads placed a fixed `label_distance` (default 7.0 Å, the mean CB→electron distance of the MTSSL 175K X1X2 rotamer library) from the anchor. When a CB distinct from CA is available the cloud is a cone (half-angle `label_cone_angle`, default 130°) about the CA→CB direction so it points away from the backbone; on coarse-grained CA-only chains it falls back to a full isotropic sphere, so the model works on both all-atom and one-bead-per-residue trajectories. Relaxation is averaged over the whole cloud *and* over frames, preserving the `r⁻⁶` non-linearity across both the conformer cloud and the ensemble. The cloud is generated on a Fibonacci lattice (no RNG), so profiles are fully reproducible.
* Steric exclusion of the label bead against the chain is always applied under `use_label=True` and can be either a `'hard'` cutoff (beads within `label_bead_radius` of any non-label CA are dropped) or a `'soft'` WCA-like repulsive wall (`w_i = exp(-Σ_j (label_bead_radius / d_ij)^label_wall_stiffness)`, reducing to the hard cutoff as the stiffness grows). The defaults (`label_steric='soft'`, `label_bead_radius=5.5` Å, `label_wall_stiffness=6.0`) were calibrated against DEER-PREdict's explicit MTSL rotamer PRE profiles across 85 disordered (trajectory) and ~2700 folded (AlphaFold) label sites; the soft wall was found to be marginally more accurate and about three times less sensitive to the bead radius than the hard cutoff, giving a single value that transfers between folded and disordered chains. This mimics the steric pruning DEER-PREdict applies to its rotamer library and stops the cloud being placed through the protein interior (critical for folded/compact chains).
* `sspre`: The residue iteration in `generate_PRE_profile` now uses the public `resid_with_CA` property rather than the private CA cache, so the PRE calculation works on coarse-grained one-bead-per-residue chains as well as all-atom trajectories. Atom coordinate lookups in the label-cloud path go through SOURSOP's memoised O(1) residue/atom lookup rather than a fresh `topology.select()` per call, keeping the calculation fast on large proteins.

### Testing
* Added tests in `tests/test_sspre.py` covering the label-cloud model: that `use_label=True` is now the default (a bare call matches the explicit label-cloud call and differs from the point model), the Fibonacci cap-direction and rotation helpers, the basic `use_label` profile (finite, bounded in [0, 1], deterministic, and correlated with the point model), the bead-radius and hard/soft steric behaviour (including convergence of the soft wall to the hard cutoff as the wall stiffens), and rejection of an invalid `label_steric` mode. The existing `test_pre_profiles_ctl9` regression against the 2019 reference profiles now pins `use_label=False`, since that reference used the point model.

## 2.0.0 (June 2026)

A major release. In addition
to wide-ranging bug fixes and behaviour-preserving speed-ups, this release adds a
consistent ensemble-reweighting (`weights`) system across the package, two new
modules for deriving frame weights from experimental data (`ssbme` and `sscoper`),
and new experimental forward-model observables (scalar couplings and NOE distances
in `ssnmr`, plus HDX protection factors in the new `sshdx` module).

### Bug fixes
* `ssprotein.get_scaling_exponent` / `get_polymer_scaled_distance_map`: Overhauled the error propagation of the log–log power-law fit. (1) The reduced chi-squared (`redchi`) is now correct and dimensionally consistent: the residual is `log(d) - (nu*log(s) + log(R0))` (an earlier fix, PR #20, addressed a missing-parenthesis/sign error and the use of linear-space `R0` instead of the log-space intercept), and the denominator is now the bootstrap standard error of each `log(RMS)` point rather than the variance of linear distances - previously a dimensionless log-space squared residual was divided by a linear-space variance (Å²), so `redchi` was meaningless. `redchi` is now a genuine reduced chi-squared (≈1 for a good fit, scaling correctly with sample size). (2) Parameter uncertainties are now a proper percentile **confidence interval** from a frame-level bootstrap (resampling frames with replacement, the same resample applied across all sequence separations), replacing the previous min/max range of disjoint data chunks - the old range was a biased estimator that grew with the `subdivision_batch_size` tuning parameter. (3) This also fixes a latent ragged-array crash for short chains / few-frame trajectories (the old chunk count could vary across separations), removes dead code (`seq_sep_RSTDS_distance`), and guards against zero-variance division. **API change:** the `subdivision_batch_size` argument is replaced by `n_bootstrap` (default 200) and a new `confidence_interval` (default 95.0); return slots 3–6 are now `nu_ci_low, nu_ci_high, A0_ci_low, A0_ci_high` (previously `min_nu, max_nu, min_A0, max_A0`). The committed regression reference pickles (`soursop/data/test_data/*_reference.pkl`) were rebuilt to capture the corrected values.
* `ssnmr`: Corrected three copy-paste errors in the propagation of Ser/Thr/Tyr neighbour-correction values to pSer/pThr/pTyr slots in the glycine-specific random-coil correction tables (`gly_ca_c`, `gly_co_c`, `gly_n_c`). Phospho-residues at the i+1 neighbour position of glycine now correctly apply the intended Kjaergaard correction; previously these slots were silently zeroed or used the wrong row, causing errors of up to ~2.7 ppm in rare Gly–phospho-Gly contexts.
* `ssprotein.get_Q`: Replaced manual sigmoid with `scipy.special.expit` to avoid `np.exp` overflow for large positive arguments.
* `ssprotein.get_Q`: Fixed a division-by-zero in residue–residue contact matrix normalisation when a pair had zero atomic contacts in all frames; affected entries are now safely set to zero rather than producing NaN.
* `ssprotein.get_contact_map`: Added an explicit check for glycine residues before computing `sidechain-heavy` contact maps; an `SSException` is now raised consistently rather than producing mdtraj-version-dependent behaviour (this resolves the historical edge case that previously caused the continuous-integration failure noted above).
* `ssprotein.get_clusters`: Fixed a division-by-zero in RMSD centroid selection when all pairwise distances within a cluster are identical (std == 0); the first frame is returned as the centroid in that degenerate case.
* `ssutils.set_numpy_threads`: Raises a descriptive `SSException` when neither MKL nor OpenBLAS is detected (e.g. macOS Apple Accelerate) rather than silently failing.
* `sstrajectory.get_interchain_contact_map` / `get_interchain_distance`: Fixed IndexError raised for terminal cap residues that lack a CA atom; added eager protein-ID and mode validation; wrapped per-residue computations in try/except so a single problematic residue no longer aborts the entire map.
* `sstrajectory.parallel_load_trjs`: Fixed a broken topology-broadcast condition that prevented a single shared topology (or a one-element list) from being applied to multiple trajectories; the argument now accepts either a `str` or a per-trajectory list and raises a descriptive `SSException` on a topology/trajectory count mismatch. This previously broke PENGUIN / `sssampling` parallel loading.
* `sstrajectory` (`__readTrajectory`): Fixed the PDB-vs-trajectory unit-cell comparison, which compared the z-dimension three times and never checked the y-dimension; the warning is now an f-string that reports the actual differing cell dimensions (and several message typos were corrected).
* `ssprotein` (initialization, `__check_cg_onebead`): No longer builds the one-letter sequence purely to obtain a residue count - that path raised `KeyError` on coarse-grained or non-standard residue names and could crash construction. It now compares the atom and residue counts directly.
* `ssprotein` (`reset_cache` / `__init__`): Consolidated the duplicated cache-initialisation logic into a single private helper so a freshly constructed object and one after `reset_cache()` are identical. Previously `reset_cache()` forced `__cg_onechain = False`, silently disabling coarse-grained handling and forcing the slow per-residue path after a reset.
* `soursop.soursop` (version helpers) and `soursop.get_version`: Fixed the version API, which still used the old Versioneer interface after the move to versioningit. `soursop/soursop.py` imported `get_versions` from `_version.py` (which versioningit writes with only `__version__`), so `import soursop.soursop` raised `ImportError`; and `soursop.get_version()` referenced an undefined `__git_revision__`, raising `NameError`. The helpers (`version`, `version_full`, `version_git_revision`) now read versioningit's `__version__` and derive the git revision from the PEP 440 local-version segment (returning `"unknown"` for a clean tagged release), with a graceful fallback to the installed-distribution metadata so the module imports cleanly even before `_version.py` is generated.

### New features and improvements
* `sstrajectory.get_interchain_contact_map` and `sstrajectory.get_interchain_distance`: Added a `stride` parameter that subsamples frames *before* the mdtraj distance computation, giving substantial speed-ups on large trajectories without any post-hoc loss of information.
* `sstrajectory`: Applied `functools.wraps` to the `lazy_loading_single_protein_trajectory` decorator so that `get_overall_radius_of_gyration`, `get_overall_hydrodynamic_radius`, and `get_overall_asphericity` preserve their docstrings and signatures at runtime (this also fixes Sphinx autodoc rendering for these three methods).
* `ssprotein`: Converted all legacy `%`-format strings to f-strings for consistency and readability.
* `ssprotein` (initialization): Replaced the per-residue `topology.select()` loop in `__get_resid_with_CA` (the documented initialization bottleneck) with a single topology pass, faithfully reproducing the original memoisation side effects so `resid_with_CA`, cap flags and `get_CA_index` results (and their dtypes) are unchanged.
* `ssprotein`: Replaced `== None` checks with `is None`; `__get_subtrajectory` now uses native mdtraj slicing (`traj[::stride]`).
* `sstrajectory`: Removed dead code and unused imports (`time`, `copy`) and the unused `single_chain_sim` value threaded through `__get_proteins`; simplified per-chain atom collection in `__get_all_proteins`.
* `sstrajectory.get_interchain_distance_map`: Hoisted the per-residue centre-of-mass computation out of the O(n1*n2) nested loop (chain-2 COMs were previously recomputed once per chain-1 residue). The non-periodic distance is now vectorised per row; the periodic path is preserved per-pair. Byte-for-byte identical output, with the redundant O(n1*n2) COM computations reduced to O(n1+n2).
* `sstrajectory.get_interchain_contact_map`: For the default `mode='atom'` (non-periodic), each residue's named-atom position is now computed once instead of via an O(n1*n2) loop of `get_interchain_distance` calls. Every other mode and the periodic path are unchanged. Output is byte-for-byte identical (verified against a pre-refactor baseline across CA/COM/atom and the untouched ca/closest-heavy/sidechain modes).
* `ssprotein` (`get_internal_scaling`, `get_scaling_exponent`, `get_local_to_global_correlation`): For `mode='CA'`, hoisted the per-residue CA centre-of-mass out of the O(n^2) sequence-separation loops via a new shared `__ca_position_cache` helper. Previously `get_inter_residue_atomic_distance` recomputed each residue's CA position from scratch on every pair (this path was *not* memoised, unlike the COM path). The helper reproduces the exact `__residue_atom_lookup -> atom_slice -> __get_subtrajectory -> 10*compute_center_of_mass` sequence, so results - including the seeded error-bootstrap / pair-resampling - are byte-for-byte identical, while the CA-position work drops from O(n^2) to O(n) (≈20-40x faster on the affected calls).
* `ssprotein.get_gyration_tensor` / `get_asphericity`: Vectorised the gyration-tensor computation across frames. The previous implementation looped over frames in Python, calling `mdtraj.compute_center_of_mass` once per frame and contracting one 3x3 second moment at a time; it now obtains the mass-weighted centre of mass for every frame in a single vectorised mdtraj call and contracts the second moment with a memory-batched `np.einsum` (the same mass-weighted COM and the same unweighted 1/N second-moment convention, so output is identical to floating-point precision). `get_asphericity` likewise replaces its per-frame `numpy.linalg.eig` loop with a single `numpy.linalg.eigvalsh` over the stacked tensors (valid because the gyration tensor is real-symmetric and the asphericity depends only on order-independent symmetric polynomials of the eigenvalues). Together this is ≈35x faster (≈230 -> ≈7,900 frames/s for a 109-residue chain). Removed the now-unused `from numpy import linalg as LA` import.
* `ssprotein`: Added two new gyration-tensor shape descriptors that reuse the (now cheap) vectorised eigendecomposition, both honouring the package-wide `weights=`/`etol` reweighting contract: `get_acylindricity` (the departure from cylindrical symmetry, `lambda_2 - lambda_1`, dimensionless `c/Rg^2` by default or raw Angstrom^2 with `normalized=False`) and `get_prolateness` (the gyration-tensor shape parameter `S = 27*prod(lambda_i - mean_lambda)/(sum lambda)^3` in `[-0.25, 2]`; `S > 0` prolate/rod-like, `S < 0` oblate/disc-like, `S = 0` spherical). Prolateness resolves a degeneracy in the asphericity: `get_asphericity` returns `delta* = 1 - 3*sum(lambda_i lambda_j)/(sum lambda)^2`, which is the relative shape anisotropy `kappa^2` and reports *how* anisotropic a conformation is but not whether it is rod- or disc-like. Validated against ideal shapes (rod `S=2`, disc `S=-0.25`, sphere `S=0`); `tests/test_ssproteins.py` gains a unit test covering the bounds and the weighted no-op contract.

### Consistent ensemble reweighting (frame `weights`)
* Every function that returns an ensemble-average value now accepts an optional per-frame `weights` vector, applied **consistently and deterministically** (closed-form weighted statistics; no stochastic resampling, fully reproducible, seed-independent). `weights=False` (the default) is an exact no-op, so all existing behaviour and the reference-observable suite are numerically unchanged.
* New shared validation/reduction helpers in `ssutils`: `validate_weights` (the single source of truth - raises `SSException` unless the vector has one finite entry per frame, every entry in `[0, 1]`, and sums to 1 within `etol`), plus `weighted_mean`, `weighted_rms`, `weighted_std` (reliability-weighted *population* estimator) and `weighted_corr`. `SSProtein.__check_weights` is now a thin wrapper around `validate_weights`.
* `stride` + `weights` is now correct: the weight vector is subsampled **and renormalised** to sum to 1 over the retained frames (previously it was silently left un-normalised).
* Converted the previously **stochastic** `numpy.random.choice` resampling in `get_internal_scaling`, `get_internal_scaling_RMS`, `get_scaling_exponent` (including its seeded error-bootstrap, now deterministically re-weighted per chunk) and `get_local_to_global_correlation` to deterministic weighted statistics, so weighting is uniform across the whole package. This also fixes a latent bug where `get_local_to_global_correlation` silently discarded user-supplied weights.
* Added the opt-in `weights` (and `etol`) parameter to the per-frame getters (`get_radius_of_gyration`, `get_hydrodynamic_radius`, `get_asphericity`, `get_end_to_end_distance`, `get_gyration_tensor`, `get_all_SASA`), to `SSTrajectory.get_overall_radius_of_gyration` / `_asphericity` / `_hydrodynamic_radius` and `get_interchain_distance_map`, and to the collapsing observables (`get_site_accessibility`, `get_regional_SASA`, `get_end_to_end_vs_rg_correlation`, `get_angle_decay`, `get_local_collapse`). When `weights` is supplied the frame axis is collapsed and the deterministic weighted statistic is returned.
* `get_internal_scaling(mean_vals=False, weights=...)` and `get_local_heterogeneity(weights=...)` raise a descriptive `SSException`: these describe a distribution / pairwise frame-vs-frame quantity for which a single per-frame probability vector is not well-defined (rather than silently returning a questionable number).
* `ssmutualinformation.calc_MI`: Fixed a latent bug (`if weights:` on a numpy array raised `ValueError: truth value ... ambiguous`) that made the weighted path of `get_dihedral_mutual_information` unusable for any real weight vector; now correctly guarded with `is not False`.
* Added the opt-in `weights` (+`etol`) parameter to five more methods so they collapse to a deterministic re-weighted ensemble value when supplied: `get_inter_residue_atomic_distance`, `get_inter_residue_COM_distance` and `get_t` (per-frame array -> scalar weighted mean; stride-aware), and `get_secondary_structure_DSSP` / `get_secondary_structure_BBSEG` **only when `return_per_frame=False`** (the per-residue fractional occupancies become frame-weighted; supplying `weights` with `return_per_frame=True` raises `SSException`, since a per-frame binary mask is not an ensemble average). Defaults (`weights=False`) are exact no-ops.
* `ssprotein.get_contact_map`: Removed the explicit "stride must be 1 if weights given" hard-block - `stride != 1` and `weights` now work together. The weight vector is validated by `ssutils.validate_weights` (one entry per trajectory frame, `len == n_frames`) and stride-subsampled/renormalised so it aligns 1:1 with the strided frames; the weighted map is the deterministic weighted mean of the per-frame contact maps. Default/unweighted behaviour unchanged.
* `ssprotein.get_Q`: Restructured into two explicit phases - (1) extract the reference frame and define the native contacts / reference distances `r0` from the **full** trajectory *first*, completely independent of stride/weights; (2) *then* resolve the analysis ensemble (apply the stride, validate the weights). This decouples `native_state_reference_frame` from the strided/re-weighted analysis and fixes a latent bug where the alignment (`superpose`) indexed the reference into the *strided* trajectory. `stride != 1` and `weights` can now be used together (previously hard-blocked). The weight contract is consistent with the rest of the package - one entry per trajectory frame (`len == n_frames`), validated by `ssutils.validate_weights` and stride-subsampled/renormalised to align 1:1 with the strided frames; the weighted `protein_average=False` breakdown is now averaged over exactly the same frame set as the unweighted path (the old code silently dropped the first frame only in the weighted branch). `protein_average=True` still (intentionally) refuses weights. Default/unweighted behaviour and the reference-observable suite are numerically unchanged.

### Reweighting against experimental data (BME / iBME and COPER / iCOPER)
* New `ssbme` module: maximum-entropy reweighting of an ensemble against experimental observables. `BME` implements Bayesian/MaxEnt reweighting (the regularised penalty `½χ² + θ·D_KL`, solved over the Lagrange-multiplier dual) with an L-curve `theta_scan` to choose `θ`, and `iBME` adds the iterative scale/offset refinement used for data with an unknown global scale (e.g. SAXS). Array-based API: `ExperimentalObservable` (with `equality`/`upper`/`lower` constraints), `BME(observables, calculated_values, initial_weights).fit(...)`, and a `BMEResult` carrying χ², the effective-frame fraction `φ`, diagnostics and a `predict()` for independent observables. Adapted from the reference Bottaro/Lindorff-Larsen BME (numpy/scipy only - no scikit-learn).
* A third variant, `BMECustom`, takes the experiment as a **raw vector + per-conformer matrix** (`(n_frames, m)`) plus an optional **custom goodness-of-fit function** `cost(experiment, calculated_values, weights) -> float` (lower = better). It is the penalty form of Bayesian Ensemble Refinement (minimise `cost(w) + θ·D_KL(w‖w0)` over the simplex via SciPy `L-BFGS-B` with a softmax reparameterisation); with no custom cost it reproduces BME's reduced χ² exactly. Useful for SAXS/PRE profiles where the cost wants to be log-scale χ², a covariance χ², Pearson correlation, etc.
* New `sscoper` module: Convex Optimization for Ensemble Reweighting (COPER; Leung et al., *JCTC* 2016). `COPER` solves the *primal* convex problem - maximise the ensemble entropy subject to a **hard** `χ² ≤ limit` constraint (no `θ`) - over the `N` frame weights via SciPy `trust-constr`, in the paper's two steps (χ²-minimisation/feasibility, then entropy maximisation). The simplex is enforced implicitly through a softmax reparameterisation (`w = softmax(z)`), so the optimiser handles only the few χ² constraints rather than `N` bound constraints plus a sum-to-one equality; this converges in tens of iterations and is hundreds to thousands of times faster than optimising the constrained weights directly (e.g. seconds rather than minutes for a 600-conformer SAXS profile), with an identical optimum. The entropy objective and χ² constraints further supply exact analytic Hessians as matrix-free `LinearOperator`s (diagonal entropy Hessian; rank-`≤ m` χ² Hessians), so no dense `N×N` matrix is formed and memory stays `O(N)` — a binding 100,000-conformer fit runs in tens of seconds where a dense Hessian would need ~80 GB. It explicitly reports **infeasibility** (when no reweighting can satisfy the data), supports **per-data-type** `χ²_α` constraints via the new `ExperimentalObservable.group` field, and reports the entropy reduction `ΔS` / mean free-energy change `⟨ΔG⟩/kT` / per-frame reweighting factors. `iCOPER` adds the iterative scale/offset refinement, and `chi2_limit_scan` is the error-scaling analogue of BME's L-curve.
* The reweighting primitives shared by both modules are factored into `ssutils` (`ExperimentalObservable`, `relative_entropy`, `weighted_linear_regression`, `find_optimal_theta`, `validate_reweighting_inputs`, `constraint_chi_squared`) and re-exported, so BME and COPER expose an identical user-facing interface. Weights produced by either plug directly into the package-wide `weights=` system above.

### Scalar (J) couplings in `ssnmr`
* `ssnmr` is no longer chemical-shifts-only: a new `compute_J3_HN_HA(protein, model="Bax2007", weights=False, ...)` computes the backbone ³J(HN, Hα) scalar coupling per frame per residue from the φ dihedral via the Karplus relation, using any of six literature parameterisations (`Bax2007`, `Bax1997`, `Ruterjans1999`, `Habeck`, `Vuister`, `Pardi`) exposed in `KARPLUS_HN_HA_COEFFICIENTS` + per-model uncertainties in `KARPLUS_HN_HA_UNCERTAINTIES`. Coefficients are ported from the [biceps package](https://github.com/vvoelz/biceps) (Voelz lab), itself adapted from MDTraj's `mdtraj/nmr/scalar_couplings.py`. A generic vectorised `karplus(angle, A, B, C, phi0=0.0)` is also exposed so any other Karplus-form relation (Bothner-By, Tvaroska, Aydin, ...) can be evaluated with the user's own coefficients. Output shape `(n_frames, n_phi)` is the natural input for `BME` / `BMECustom` / `COPER`; the function honours the package-wide `stride` + `weights` reweighting contract, and a `return_uncertainty=True` flag returns the model's forward-model σ in Hz alongside the J array.

### NOE distances in `ssnmr`
* New `compute_NOE_distances(protein, atom_pairs, stride=1)` returns the per-frame `(n_frames, n_pairs)` inter-atom distance matrix in Å for an arbitrary list of atom pairs (thin wrapper around `mdtraj.compute_distances` with the soursop unit + shape convention). A companion `noe_ensemble_average(distances, power=6, weights=False, ...)` collapses such an array via the NOE `(Σ w_i d_i⁻ᵖ)^(-1/p)` rule (defaults to `p=6`; `p=3` supported), honouring the package-wide `weights=` contract. The per-frame matrix (raised to the `-p` power) is BME/COPER-ready against an experimental NOE profile.

### HDX protection factors (new `sshdx` module)
* New `sshdx.compute_protection_factors(protein, beta_c=0.35, beta_h=2.0, beta_0=0.0, ...)` predicts per-residue ln(P) for backbone amide H/D exchange via the **Best–Vendruscolo** formula `lnP_i = β_c·N_c(i) + β_h·N_h(i) + β_0`. `compute_Nc` counts heavy atoms within 6.5 Å of the amide N of each residue (excluding `|i-j| ≤ 2`); `compute_Nh` counts backbone H-bonds the amide H of residue `i` forms (per-frame, via `mdtraj.wernet_nilsson`). All three helpers return `(residue_indices, (n_frames, n_residues) array)` so the per-residue mapping is explicit; the `(n_frames, n_residues)` shape is the natural input for BME / COPER reweighting against experimental HDX protection factors. Proline (no backbone H) and any residue lacking a recognisable backbone amide H are dropped; the package-wide `stride` + `weights` contract is honoured.

## Feb 2024 (0.2.5)
* Removed periodic correction in SSProtein functions. We could provide periodic boundary condition checks for some but not all functions, and ensuring this flag was possible everywhere is not feasable. With this in mind, we opted to make a design decision to remove the `periodic` flag from the small number of functions that had it, such that there's no risk of a user forgetting and analyzing two distinct properties for a trajectory that requires PBC fixing where one analysis used `periodic=True` whereas another did not have this option. Now, the user must ensure their protein trajectories are corrected ahead of time. Note that we also added support for periodic=True for some of the SSTrajectory analysis functions, because in cases where multiple chains are present reconstructing a non-PBC trajectory becomes much more difficult. As such, intermolecular analysis does provide intrinsic PBC correction, whereas intramolecular analysis does not.

## July 2022 (0.2.1)
For version 0.2.1 we introduce potentially breaking changes into how COM distances are reported.

**Details:** All center-of-mass (COM)-based functions now return distances (as before) and relative positions in x/y/z (this is new) in Angstroms, not nanometers. Previously, the various center-of-mass based functions that returned absolute positions (i.e. 3xn matrices of x/y/z vs frame number) returned them where x/y/z was in units nm. This is fine if you know this, but means if you manually calculate distance between two COM vectors you'd get a distance in nanometers and not Angstroms. This is an unhelpful and unexpected behavior given all other distances are in Angstroms, so we have made the decision to fully update to Angstroms even for vector positions. This DOES NOT break any code internal to soursop, but if you were using COM positions to manually calculate distances these may need to be recalculated.

## July 2021
CAMPARITraj is SOURSOP! For the final release we have re-named and re-branded CAMPARITraj as SOURSOP. This change in the name serves two important purposes.

Firstly, CAMPARITraj was borne out of a collection of scripts and code built to work with CAMPARI. However, it has evolved into a stand-alone package for the analysis of all-atom simulations of IDPs and IDRs, and importantly, much of the analysis it performs can also be done in CAMPARI. As such, we felt it was important to decoupled SOURSOP from CAMPARI, both to avoid the implication that CAMPARI cannot perform analyses itself, and to avoid a scenario in which it may appear that this package only works with CAMPARI simulations.

Secondly, there is a long, rich tradition of naming software tools after drinks in the Pappu lab (CAMPARI, ABSINTH, CIDER, LASSI etc.). As such SOURSOP is first-author Jared's Caribbean twist on this theme!

> **WARNING (historical):** We are currently and systematically updating all of the CAMPARITraj codebase to SOURSOP. As such, for now, we recommend using a previous version. Note the SOURSOP change breaks all backwards compatibility. Sorry about that.
